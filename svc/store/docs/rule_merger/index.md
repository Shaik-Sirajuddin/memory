# Rule Merger Component

The `RuleMerger` is a critical background component designed for the `svc/store` service. Its primary responsibility is to orchestrate the consolidation of temporary instruction branches (generated by agents) into a single, cohesive main file version.

## Purpose

In automated agent systems acting on files, multiple concurrent edits or staggered updates against a core "instruction" file can cause race conditions or diverging branches. The `RuleMerger` solves this by:
1.  **Deduplicating and Debouncing**: Using a 10-second cache-backed sliding window to prevent redundant processing of the same file.
2.  **Temporal Ordering**: Consuming branch files written with unix timestamps and applying them sequentially.
3.  **LLM-Assisted Deduplication**: Iterating over the modified sequence of instructions and invoking an external LLM agent to intelligently compress or deduplicate adjacent, similar rules.
4.  **Race Prevention**: Leveraging OS-level file locking (`syscall.Flock`) to ensure the main file cannot be corrupted during the merge cycle.

## Architecture

The `RuleMerger` operates asynchronously. The store service interacts with it primarily by enqueuing file IDs (paths).

*   **Enqueuing**: When a store component registers an update to a temporary branch file, it calls `RuleMerger.Enqueue()`.
*   **Cooldown**: The queue verifies against a Redis cache. Multiple enqueues for the same file within a 10-second window are safely dropped.
*   **Processing**: A background worker continuously drains the queue. When it processes a file, it implements a 10s cooldown per file based on the last processed time to prevent thrashing.
*   **Event Emission**: After a successful file lock and rewrite, it triggers the `EventBus` to notify the rest of the system about exactly which sequenced lines were perturbed.
